BioVerify Documentation

BioVerify is a biometric identity verification platform. It uses ISO/IEC 19794-2 fingerprint templates to authenticate individuals at the point of service — pharmacy dispensing, clinic check-in, financial account access, and more.

BioVerify is available as a standalone module. It does not require GENFIN membership infrastructure — any organisation can integrate it independently via the REST API or JavaScript SDK.

How it works

A USB fingerprint reader at the terminal captures a live scan. The BioVerify SDK converts it to a normalised template and sends it to the BioVerify API for 1:1 matching against the enrolled template for the given member ID. The match result (with a confidence score) is returned in under 2 seconds.

Quickstart

The fastest way to get a working verification flow is the JavaScript SDK. Add the script tag, initialise with your API key, and call bv.verify().

HTML
<script src="https://cdn.bioverify.io/sdk/v2/bv.min.js"></script>
JavaScript
const bv = BioVerify.init({
  apiKey: 'bv_live_xxxxxxxxxxxx',
  terminalId: 'TERM-0041',
  mode: 'pharmacy_dispense'
});

const result = await bv.verify({
  memberId: 'GF-0041882',
  context: 'pharmacy_dispense',
  rxId: 'RX-889244'
});

if (result.matched) {
  console.log('Identity confirmed — dispense authorised');
  console.log('Confidence:', result.confidence); // e.g. 0.998
} else {
  console.warn('Match failed:', result.reason);
  // reason: 'no_enrolment' | 'low_confidence' | 'sensor_error'
}
In test mode (API key beginning bv_test_), any 6-digit member ID returns a successful match so you can develop without a physical reader.

Authentication

All API requests must include your API key in the Authorization header as a Bearer token.

HTTP header
Authorization: Bearer bv_live_xxxxxxxxxxxx

API keys are scoped to a single site. Use a separate key per terminal group or organisation for auditability. Keys are available from the Admin Console under Settings → API keys.

SDK — Installation

BioVerify ships as a UMD bundle (browser), a CommonJS module (Node/Electron), and an ES module.

CDN (browser)
<script src="https://cdn.bioverify.io/sdk/v2/bv.min.js"></script>
npm / yarn
npm install @bioverify/sdk

# or
yarn add @bioverify/sdk
ES module import
import BioVerify from '@bioverify/sdk';

SDK — Initialise

BioVerify.init(config) returns a configured bv instance. Call once at application start.

SDK — bv.verify()

Triggers a fingerprint scan on the connected reader and matches the result against the stored template for the given member.

Signature
await bv.verify(options: VerifyOptions): Promise<VerifyResult>

VerifyOptions

memberId required
string
The member's unique identifier (e.g. GENFIN member number or your own user ID).
context
string
Overrides the instance-level mode for this individual verification.
rxId
string
For pharmacy use: the prescription ID being dispensed. Stored in the audit log.

VerifyResult

matched
boolean
True if the fingerprint matched the enrolled template above the confidence threshold (default 0.95).
confidence
number
Match score from 0.0 to 1.0.
verificationId
string
Unique ID for this verification event. Use for audit trail lookups.
reason
string
Set when matched is false. One of: no_enrolment, low_confidence, sensor_error, timeout.
durationMs
number
Total time from scan trigger to result, in milliseconds.

SDK — bv.enrol()

Registers a new fingerprint template for a member. Captures three scans, derives a composite template, and stores it securely against the member ID.

JavaScript
const result = await bv.enrol({
  memberId: 'GF-0041882',
  finger: 'right_index', // optional, for documentation
  operator: 'admin@ambuya.co.zw'
});

if (result.enrolled) {
  console.log('Template stored. Quality score:', result.quality);
}
Enrolment requires informed consent under GDPR (Art. 9) and equivalent legislation. You must record the member's explicit consent before calling bv.enrol(). See GDPR & compliance.

SDK — Events

The bv instance is an event emitter. Subscribe to events for UI feedback during the scan flow.

JavaScript
bv.on('scan:start', () => showScanner());
bv.on('scan:progress', ({ percent }) => updateProgress(percent));
bv.on('scan:matched', ({ confidence }) => showSuccess(confidence));
bv.on('scan:failed', ({ reason }) => showError(reason));
bv.on('sensor:disconnected', () => showSensorWarning());

REST API — POST /verify

Base URL: https://api.bioverify.io/v2

For environments where the JavaScript SDK isn't suitable (server-side integrations, Electron apps using native USB libs), use the REST API directly by sending the fingerprint template as a Base64-encoded ISO 19794-2 record.

POST /verify Match a fingerprint template
Request body (JSON)
{
  "memberId": "GF-0041882",
  "template": "Rk1SAAAA...", // Base64 ISO 19794-2 template
  "context": "pharmacy_dispense",
  "terminalId": "TERM-0041",
  "rxId": "RX-889244"
}
Response (200 OK)
{
  "matched": true,
  "confidence": 0.998,
  "verificationId": "vrf_01j3kpq8xz...",
  "durationMs": 1620,
  "member": {
    "id": "GF-0041882",
    "name": "Chiedza Nyamuzihwa",
    "enrolled": true
  }
}

REST API — POST /enrol

POST /enrol Store a fingerprint template for a member
Request body
{
  "memberId": "GF-0041882",
  "templates": [
    "Rk1SAAAA...", // scan 1
    "Rk1SAAAA...", // scan 2
    "Rk1SAAAA..."  // scan 3
  ],
  "finger": "right_index",
  "consentRef": "consent_doc_20250610_gf0041882",
  "operator": "admin@ambuya.co.zw"
}

REST API — GET /members

GET /members List enrolled members
GET /members/:memberId Get enrolment status for one member
DELETE /members/:memberId/template Delete stored biometric data (right to erasure)

REST API — GET /terminals

GET /terminals List registered terminals and status
GET /terminals/:terminalId Get heartbeat, firmware version, last-seen timestamp

REST API — GET /logs

GET /logs Query verification audit log

Query parameters: from, to (ISO 8601 dates), terminalId, memberId, result (matched | failed), limit (default 50, max 1000), cursor (pagination).

Webhooks

BioVerify can POST event payloads to your endpoint in real time. Configure webhook URLs in the Admin Console under Settings → Webhooks.

Supported events

  • verification.matched — successful identity match
  • verification.failed — match failed or escalated
  • member.enrolled — new template stored
  • member.template_deleted — biometric data erased
  • terminal.offline — terminal heartbeat missed for 5 minutes
Validate webhook signatures using the BioVerify-Signature header (HMAC-SHA256 of the raw body, signed with your webhook secret).

GDPR & compliance

Fingerprint templates are special category biometric data under GDPR Article 9 and equivalent legislation (Zimbabwe Data Protection Act, POPIA in South Africa). Operators must:

  • Obtain explicit, written consent before enrolment and record the consent reference in the consentRef field
  • Inform members of the purpose, retention period, and their right to erasure
  • Honour erasure requests using DELETE /members/:memberId/template within 30 days
  • Not use BioVerify data for purposes other than identity verification at the point of service

BioVerify stores only the derived ISO 19794-2 minutiae template — not the raw fingerprint image. Templates are encrypted at rest (AES-256) and in transit (TLS 1.3). Retention defaults to 7 years for regulated healthcare environments; contact support to configure custom retention policies.

A Data Processing Agreement (DPA) is available on request. Required for GDPR compliance when BioVerify processes biometric data on behalf of a data controller.
